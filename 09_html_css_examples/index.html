Exemple practice HTML + CSS (nav, forms)

Scopul: să explicăm cum să construiești un navbar accesibil și un formular practic, folosind HTML semantic, CSS (Flexbox/Grid minimal) și validare JS.

Partea teoretică

Definiții & concepte-cheie:
  • <nav> — element semantic pentru navigație. Reprezintă o zonă care conține linkuri către alte părți ale site-ului sau paginii.
    ✅ Exemplu: 
    <nav aria-label="Navigație principală">
  <a href="#acasa">Acasă</a>
  <a href="#contact">Contact</a>
</nav>
    Acest exemplu arată o navigație semantică și accesibilă prin aria-label, care descrie scopul zonei.
  • <form> este containerul pentru câmpurile de input și permite trimiterea datelor către un server. Este important să asociezi fiecare câmp cu un <label> pentru accesibilitate: 
    <form>
  <label for="email">Email</label>
  <input id="email" name="email" type="email" required>

  <button type="submit">Trimite</button>
</form>
    Această asociere permite cititoarelor de ecran să anunțe corect scopul câmpului. De exemplu: **„Email — câmp de editare, obligatoriu”**. Atributul required activează validarea automată în browser și indică faptul că acel câmp trebuie completat pentru a trimite formularul.
  • aria-* — set de atribute care adaugă informații suplimentare pentru accesibilitate (ARIA = Accessible Rich Internet Applications).
    • aria-label — oferă o descriere clară (ex: <button aria-label="Deschide meniul">)
    • aria-expanded — indică dacă un meniu este deschis (true) sau închis (false).
    • aria-live="polite" — anunță mesajele care se schimbă dinamic (ex: “Mesaj trimis cu succes”).
    ⚙️ Atributele aria-* sunt folosite doar când HTML-ul nu este suficient de descriptiv. Nu înlocuiesc elementele semantice (ex: <nav>, <button>, <header>), ci le completează.
Diagrame conceptuale:

🔹 Relația dintre HTML, CSS și ARIA în construirea unui navbar accesibil: 


[ HTML semantic ]
       ↓
<nav aria-label="Navigație principală">
   <button aria-expanded="false">☰</button>
   <ul><li><a>Link</a></li></ul>
</nav>

[ CSS ]
→ Flexbox/Grid pentru structură și responsive
→ Media queries pentru mobil

[ ARIA ]
→ clarifică scopul elementelor pentru screen reader
→ actualizat dinamic cu JS (ex: aria-expanded)
Comenzi / API-uri utile:
  • <nav>, <ul>, <li> pentru navigație
  • <form>, <label>, <input>, required, type="email" pentru formulare
  • CSS Flexbox: display: flex; align-items: center; gap: 1rem;
  • JavaScript: getAttribute(), setAttribute() pentru actualizarea atributelor aria-*
Bune practici:
  • Folosește etichete clare și mesaje de eroare vizibile.
  • Asociază fiecare <label> cu câmpul său folosind for și id.
  • Menține tabindex natural — evită forțarea ordinii cu valori numerice, cu excepții bine justificate.
  • Folosește aria-label doar când nu există text vizibil (ex: iconițe fără text).
  • Adaugă atribute precum aria-required și aria-invalid când gestionezi validarea cu JavaScript.
  • Grupează câmpuri relevante cu <fieldset> și folosește <legend> pentru a descrie scopul grupului.
  • Nu abuza de stiluri inline — definește clase CSS reutilizabile și semantice pentru consistență și întreținere ușoară.

Exemplu 1 — Navbar simplu (HTML + CSS)

Descriere: navbar responsive, cu buton pentru mobile (fără librării). 

<nav class="site-nav">
  <a class="brand" href="#">Logo</a>
  <button class="nav-toggle" aria-expanded="false">☰</button>
  <ul class="nav-list">
    <li><a href="#home">Acasă</a></li>
    <li><a href="#servicii">Servicii</a></li>
    <li><a href="#contact">Contact</a></li>
  </ul>
</nav>

Rezolvare: folosim CSS pentru a ascunde lista pe mobil și JS pentru a comuta aria-expanded. 

<style>
.site-nav { display:flex; align-items:center; gap:1rem; }
.nav-list { display:flex; list-style:none; gap:0.6rem; }
@media (max-width:600px){
  .nav-list { display:none; flex-direction:column; }
  .nav-list.open { display:flex; }
}
</style>

Exemplu 2 — Formular de contact (HTML + CSS + JS)

Descriere: formular simplu cu validare minimă JS și mesaje inline. 

<form id="contactForm">
  <label for="name">Nume</label>
  <input id="name" name="name" required />

  <label for="email">Email</label>
  <input id="email" name="email" type="email" required />

  <label for="message">Mesaj</label>
  <textarea id="message" name="message" required></textarea>

  <button type="submit">Trimite</button>
</form>

Rezolvare: validare JS, afișare mesaj succes/eroare, prevenire submit default pentru demo. 

document.getElementById('contactForm').addEventListener('submit', function(e){
  e.preventDefault();
  // validare simplă
  const email = this.email.value;
  if(!/.+@.+\..+/.test(email)) { /* arată eroare */ }
  else { /* arată succes */ }
});
Exerciții practice:
  1. Creează un navbar similar celui de mai sus și fă-l responsive.
  2. Adaugă un formular de contact care arată mesaje inline la validare (vezi partea de exemple).
  3. Extensie: salvează datele în localStorage și afișează-le într-o listă sub formular.

    Întrebări de interviu

    • Ce element semantic folosești pentru a declara o zonă de navigare și de ce?

      <nav> — este un element semantic HTML folosit pentru a grupa linkuri de navigație principală. Ajută la accesibilitate (screen reader-ele știu că e un meniu) și structurarea clară a paginii.

    • De ce este importantă asocierea <label> cu input?

      Asocierea prin for (sau înglobare) permite screen reader-elor să citească corect câmpul. Crește accesibilitatea și ușurează selectarea câmpului la click pe etichetă.

    • Cum faci un navbar responsive fără framework?

      Folosind HTML semantic (<nav>), CSS Flexbox și @media queries pentru a ascunde/afișa meniul pe mobil. JS poate fi folosit pentru a comuta clasele (ex: open).

    • Ce înseamnă novalidate pe un form?

      Dezactivează validarea HTML5 nativă a browserului. Este util când vrei să controlezi validarea complet cu JavaScript.

    • Cum oferi feedback vizual pentru erori de validare? (ARIA + vizual)

      Se poate folosi aria-live="polite" pentru mesaje și clase CSS precum .error cu :invalid pentru stiluri (roșu, borduri, mesaje inline).

    • Care sunt avantajele validării pe client vs. server?

      Client: rapid, interactiv, reduce cererile. Server: sigur, nu poate fi evitat. Ideal: ambele.

    • Ce este aria-expanded și când îl folosești?

      Atribut folosit pe elemente interactive (ex: buton) pentru a indica dacă o zonă e deschisă (true) sau închisă (false). Ajută utilizatorii de screen reader.

    • Cum previi trimiterea repetată a formularului (double submit)?

      Dezactivezi butonul de submit după primul click, sau folosești un flag în JavaScript pentru a ignora trimiterea ulterioară.

    • Explică cum folosești localStorage pentru salvare temporară a formularelor.

      La submit sau click pe buton, colectezi valorile și le salvezi cu localStorage.setItem("cheie", JSON.stringify(...)). Le poți citi apoi cu getItem și le afișezi.

    • Ce trebuie să verifici când faci un design pentru mobil (mobile-first)?

      Fonturi lizibile, layout vertical, butoane mari și spațiate, meniuri ascunse, media queries pentru ecrane mici (max-width), testare pe touch.

    • Diferența între <button> (implicit) și <input type="submit">.

      button e mai flexibil (poate conține HTML, text, SVG). Ambele pot trimite formularul, dar button poate avea și type="button" sau reset.

    • Cum testezi accesibilitatea unui formular (screen reader, tab, color contrast)?

      Testezi navigarea cu Tab, verifici contrastul (ex. WCAG), asculți cu un screen reader (NVDA, VoiceOver), folosești label corect și aria-atribute dacă trebuie.

    • Ce face required pe un input și ce limitări are?

      Forțează completarea câmpului înainte de submit (în HTML5). Limitare: poate fi evitat cu JS, deci trebuie validare și pe server.

    • Ce rol are autocomplete pe input-uri?

      Ajută browserul să sugereze completări pe baza valorilor introduse anterior (ex: email, nume). Crește experiența utilizatorului.

    • Explică diferența între placeholder și label.

      label e vizibil tot timpul și accesibil, placeholder dispare când scrii și nu e citit mereu de screen reader. Nu le folosi unul în locul altuia.

    • Ce sunt pseudo-clasele :focus și :invalid și cum le folosești?

      :focus stilizează un input activ, :invalid marchează un câmp invalid conform validării HTML. Le folosești pentru feedback vizual.

    • Cum faci un layout în care navbar-ul rămâne fix în partea de sus (sticky) fără JS?

      Folosind CSS: position: sticky; top: 0; pe navbar. Asigură-te că are un z-index mare și un background setat.
